B5. Add #378 Industrial Throughput (new endless) needs reframe · may need program change

⚠ Reframed per Joseph's clarification + Xcode review.

Terminology Joseph confirmed (2026-07-09): in Star Atlas, a "processor" is a claim-stake building that takes an input and produces an output (e.g. copper ore → copper). An extractor is a claim-stake building that produces output only (e.g. copper ore). Crafting habs are different — not part of a claim stake, manual crafting only.

The crafting_efficiency_modifier field on CharacterModifier (research.rs:21) is consumed only at start_crafting_process.rs:243 — that's crafting habs. No CharacterModifier field affects claim-stake processor throughput. So "Industrial Throughput" as originally worded ("processor efficiency") cannot be authored in config — it needs a new program field.

Two paths — pick one:

PathWhat it doesCostPlayer fantasy
B5a — Crafting-hab efficiency (config-only) Uses existing crafting_efficiency_modifier. Boosts crafting-hab output only. 0 program work "My crafting habs produce +10% more at L10."
B5b — Claim-stake processor throughput (needs program change) New field on CharacterModifier + consumer in claim-stake processing path. Same lift as level_penalty_bps (~<1 day per Jacob). ~<1 day program + account migration "My copper-ore → copper processors produce +10% more at L10."

compute concern Endless nodes + combine_repeated = CU blowout

⚠ Verified program-side hazard (Xcode review, 2026-07-09).

character.rs:82-84 — during sync_with_tree, for every unlocked node, the engine calls combine_repeated(&node_def.modifier, level). And research.rs:132-137 is literally:

pub fn combine_repeated(&mut self, other: &CharacterModifier, times: u32) -> Result<()> {
    for _ in 0..times { self.combine(other)?; }  // ← O(level) loop
    Ok(())
}

So a character with a level-500 endless node triggers 500 combine() passes per sync, and sync runs on every instruction (unless seq_id matches, but re-sync happens after every tree bump). That's a compute-unit blowout that can effectively brick high-level characters — and it compounds across multiple endless nodes.

Recommended program-side hardening (Xcode): Add a delta × level fast path in combine_repeated so the multiply is constant-cost instead of loop-cost. Also add aggregate clamps (max-level cap per node). This is the same class of program-change lift as level_penalty_bps and should batch with it. Endless nodes should not ship at scale without this fix.

Monday path: if we want to validate the endless-node pattern with real players, we can ship #82 activated with a low practical ceiling (say, L20-L50) and document that higher levels require the combine_repeated fast path. The fast-path work is the gating dependency for unbounded endless nodes.
